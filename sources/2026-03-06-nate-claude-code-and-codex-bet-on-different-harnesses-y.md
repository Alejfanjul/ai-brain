The harness is the product When you use an AI coding agent, Claude Code, Codex, Cursor, any of them, you’re interacting with two things at once. There’s the model: the intelligence, the part that understands your request and generates code. That’s the part everyone compares. That’s what the benchmarks measure.  Then there’s everything else. Where does the AI actually do its work, on your computer, or on a server somewhere? And it matters more than the model, because the model determines how smart your AI is, but the harness determines how it fits into your work. How it collaborates with you. What it can touch, what it remembers, how it fails, what happens when you want to switch to a different tool in six months. The harness is the relationship. The model is just the brain inside it.  Nobody compares harnesses. Every comparison you’ve read this month, Claude vs. GPT, Opus vs. Codex, is comparing brains. But the thing that actually determines whether an AI coding agent makes your team faster or drives them crazy, whether you can change tools next year or you’re stuck, that’s all harness. And it’s invisible in every evaluation that matters.  This would be a minor blind spot if the harnesses were all roughly the same. They’re not. They’re diverging. Fast, and on purpose. Claude Code and Codex aren’t two flavors of the same thing. They embody fundamentally different ideas about how humans and AI should work together. One sits in your actual workspace, with access to everything on your machine, building up memory of your project over time, talking through its reasoning as it goes. The other works in a sealed room with a copy of your code, thinks privately, and slides finished results under the door. One is a collaborator at the desk next to yours. The other is a contractor in a clean room.  The difference is philosophical before it’s technical, and your team is building around whichever philosophy you chose. The habits, the processes, the verification steps, the integration plumbing, all of it accumulates around whichever harness you chose, gaining value every month. Switch harnesses and all of that resets to zero. That’s the lock-in nobody is pricing into their decisions: not lock-in to a vendor’s subscription, but lock-in to a vendor’s philosophy of how work should happen.  On February 5, Anthropic and OpenAI both released new flagship coding models on the same day: Claude Opus 4.6 and GPT-5.3-Codex. I’ve been tracking the developer community’s response for three weeks. The models are converging on capability. The harnesses are diverging on everything else. That divergence is the story, and the window for choosing deliberately is closing, because every week your team spends inside one harness is another week of compounding dependency.  Here’s what that divergence actually looks like, what it costs, and why this is the most consequential tool decision in software right now. Proof the harness matters more Before we go further, one number that makes the thesis concrete. At the AI Engineer Summit in January 2026, Anthropic presented results from the CORE benchmark, which tests agents’ ability to reproduce published scientific results. The same Claude Opus 4.5 model, identical weights, identical training, scored 78% when running inside Claude Code’s harness. It scored 42% when running inside Smolagents, a different harness. Same brain. Different body. Nearly double the performance.  That’s not a marginal difference explained by prompt engineering. It’s a structural difference explained by everything the harness does: how it manages context, how it hands off state between sessions, how it connects tools, how it verifies results. The harness isn’t an optimization layer on top of the model. It’s a performance multiplier that determines whether the model’s intelligence actually translates into useful work.  I’ve been running both of these tools daily since launch, and the benchmark confirms what I kept feeling in practice: the same prompt, the same intent, lands completely differently depending on which harness is running it. Now: the two harnesses that matter most right now are making very different bets about what that body should look like. Two architectures, not two flavors Anthropic’s engineering team published a detailed account of the problem their harness was built to solve. They framed it vividly: imagine a software project staffed by engineers working in shifts, where each new engineer arrives with no memory of what happened on the previous shift. That’s what happens when an AI agent works across multiple context windows. The model is smart, but it starts each session blank.  Their solution was structural, not just prompting. Claude Code’s harness uses a two-part pattern: an initializer agent that runs first to set up the project, creating a structured feature list, an init script, a progress log, and a clean git commit. Then a coding agent runs in every subsequent session, making incremental progress on one feature at a time and leaving structured artifacts for the next session. The progress file and git history become the agent’s institutional memory. Every session begins the same way: read the progress log, check the git history, run the basic tests to confirm nothing is broken, then pick the next feature and start.  The key design choice is that the harness forces incrementalism. Left to its own devices, the model tries to build everything at once (Anthropic calls this “one-shotting”) and runs out of context mid-implementation, leaving the next session to guess at half-finished work. The harness prevents this by structuring the task list as JSON (not markdown, because the model is less likely to corrupt structured data formats) and prompting the agent to work on exactly one feature per session. The harness also forces verification: the agent uses browser automation tools like the Puppeteer MCP server to test features end-to-end the way a human would, catching bugs that unit tests miss.  This is the architecture beneath the surface of Claude Code. It runs in your actual terminal: your shell, your environment variables, your SSH keys, your toolchain. Anthropic’s engineers describe this philosophy as “bash is all you need”: rather than building dozens of specialized tools, the agent uses composable Unix primitives like grep, git, and npm, chaining them together. This keeps the context window lean (tool descriptions are expensive) and gives the agent access to everything a human engineer would have. The tradeoff is that the trust boundary is your entire workstation.  OpenAI’s harness engineering team arrived at a different architecture from a different starting point. They published a detailed account of building a million-line internal product over five months using only Codex agents, zero lines of manually-written code, roughly 1,500 pull requests, initially driven by three engineers. Their central insight was almost the opposite of what you’d expect: early progress was slower than anticipated, not because Codex couldn’t write code, but because the environment was underspecified. The agent lacked the structure, tools, and feedback mechanisms to make progress toward high-level goals.  Their response was to make the repository the system of record for everything. Architecture decisions, alignment threads, product principles, anything not in the repo was “illegible” to the agent and therefore didn’t exist. They tried the “one big AGENTS.md” approach and it failed: when everything is marked as important, nothing is, and the file “rots instantly” into a graveyard of stale rules. Instead, they built a progressive disclosure system of focused, cross-linked documentation that the agent could navigate. They enforced a rigid layered architecture with validated dependency directions and limited permissible edges, all checked by custom linters, which were themselves written by Codex. The linter error messages doubled as remediation instructions, so when the agent violated an architectural rule, the error told it exactly how to fix the violation.  This is the architecture beneath the surface of Codex. It runs tasks in isolated cloud containers, your code is cloned in, internet access is disabled by default, and the agent works independently. Where Claude Code gives the agent full access to your environment and manages the risk through incrementalism and human oversight, Codex constrains the agent’s environment and manages the risk through isolation and mechanical enforcement. Where Anthropic’s harness makes the agent remember, OpenAI’s makes the codebase remember. Both solve the same problem: how do you get reliable work from an AI across many sessions, through genuinely different theories of where institutional knowledge should live?  Calvin French-Owen, who helped launch the Codex web product and now uses both tools extensively, describes the practical result: he picks his coding agent as a function of how much time he has and how long he wants it to run autonomously. Claude Code for planning, orchestrating his terminal, and explaining how parts of the codebase work. Opus spins up sub-agents simultaneously, delegates exploration to fast Haiku instances, and is “more creative” in terms of suggesting things the developer forgot to mention. Codex for the actual code, “the Codex code just straight up has fewer bugs.” He starts with Claude Code and keeps it open, then flips to Codex when he’s ready to implement. Every so often, he has Codex review Claude’s work, and it catches mistakes Claude missed. These aren’t interchangeable tools. They’re complementary architectures that reward different kinds of investment. Five points of divergence The architectural gap between these platforms isn’t one thing. It’s five things, all compounding simultaneously. And the primary sources from both companies reveal how deliberate, and how differently motivated, each choice is.  Execution philosophy. Anthropic’s position is that “bash is all you need.” Rather than building specialized tools with long descriptions that eat up the context window, Claude Code gives the agent access to composable Unix primitives, grep, git, npm, ffmpeg, the GitHub CLI, and lets it chain them together with pipes. A single line of bash can query a database, filter results, and write them to a file. This is cheaper in tokens than calling three separate tools, and more flexible. The ML6 team’s analysis showed that the GitHub MCP server’s 38 tools consume 15,000 tokens of tool descriptions; the GitHub CLI achieves the same functionality with far fewer tokens in the context window.  OpenAI wired Chrome DevTools Protocol directly into the Codex agent runtime, giving it access to DOM snapshots, screenshots, and navigation capabilities so it could reproduce UI bugs and validate fixes by actually driving the application. They also gave each agent its own ephemeral observability stack: Victoria Logs and Victoria Metrics spin up per git worktree and disappear when the work is done, letting the agent query logs and metrics directly. A prompt like “make the service start in under 800ms” becomes a testable acceptance criterion because the agent can actually measure startup time. If the agent can’t measure it, it can’t improve it.  Both philosophies give the agent hands. But one gives it your hands — full access to your actual environment, composable and powerful and exactly as dangerous as it sounds. The other builds it custom hands in a controlled room, safer by default but less able to reach the tools you already use.  State and memory. The Anthropic harness solves the cross-session memory problem with structured artifacts. Their engineering report describes a progress file (claude-progress.txt) that every coding agent reads at the start of a session and updates at the end, plus a feature list stored as JSON, deliberately not markdown, because the model is less likely to inappropriately modify structured JSON than freeform text. These files, combined with descriptive git commits, create a trail that any new agent instance can follow to understand where the project stands and what to do next. Developers who invest in these artifacts, CLAUDE.md files for project conventions, progress logs for session continuity, feature lists for task tracking, build a compounding asset. The more context accumulates, the better every subsequent session works.  OpenAI’s approach pushes institutional memory into the repository itself. Anything not in the repo is illegible to the agent and therefore doesn’t exist. Architecture decisions, product principles, team alignment, all of it gets encoded as documentation, linter rules, or structural tests. When the agent violates an architectural boundary, the linter error message tells it exactly how to fix the violation, turning enforcement into instruction. They also discovered an entropy problem unique to agent-generated code: Codex replicates whatever patterns exist in the repository, including uneven or suboptimal ones, and this inevitably leads to drift. Their initial response, spending every Friday (20% of engineering time) manually cleaning up what they called “AI slop,” didn’t scale. So they encoded “golden principles” into the repository and built automated cleanup processes where background Codex tasks scan for deviations and open targeted refactoring PRs. The repo polices itself.  One harness makes the agent remember. The other makes the codebase remember. Both work. Neither transfers cleanly to the other, because all the investment your team makes in CLAUDE.md files is worthless in Codex, and all the investment in linter-as-instructor architecture is worthless in Claude Code.  Context management. Both companies learned the same lesson about context: more isn’t better if it’s not curated. OpenAI tried the “one big AGENTS.md” approach, a monolithic instruction file for the agent, and it failed in three ways. It crowded out the actual task and code. When everything was marked important, nothing was. And it rotted instantly as humans stopped maintaining it. Their solution was progressive disclosure: focused, cross-linked documentation files that the agent navigates to find what it needs, when it needs it.  Anthropic arrived at a similar principle from a different direction. Rather than loading all available tool descriptions into the system prompt at the start (which wastes context before the agent does any work), Claude Code stores tools and skills as files on the filesystem and lets the agent retrieve them just-in-time based on the incoming request. A “tool search tool” lets the agent semantically search available capabilities instead of having them all pre-loaded. Calvin French-Owen, writing from heavy use of both platforms, puts the underlying principle simply: “your work needs to somehow be chunked” to fit context windows, “compaction is a lossy technique,” and you should “stay in the smart half of the context window” because model performance degrades as the window fills up.  The practical divergence is that Claude Code manages context through compaction and sub-agent delegation, automatically summarizing older context and spinning up parallel agents that each get their own window. Codex manages context through isolation; each task runs in its own sandbox with clean context, so tasks don’t compete for space. Claude is better when one task needs deep understanding of a massive codebase. Codex is better when you’re running many independent tasks in parallel.  Tool integration. Anthropic created MCP, the Model Context Protocol, which is now the open standard for connecting AI agents to external tools. It’s now backed by OpenAI, Google, Microsoft, and governed by the Linux Foundation. Claude Code was built around MCP from the start. But the more interesting harness insight is how both companies handle the cost of tool integration inside the context window.  Anthropic introduced Skills, markdown files and scripts stored on the filesystem. The agent only sees short names and descriptions of each skill (50-100 tokens), not the full instructions (which could be thousands). It reads the full skill definition only when it decides to use one. This is context management as harness design: the tool integration layer is deliberately architected to be stingy with tokens.  OpenAI’s Codex App Server takes a different approach: a bidirectional JSON-RPC harness that runs alongside your stack and exposes tools, git, test runners, Chrome DevTools, app logs, metrics, as RPC endpoints. The agent calls into those tools programmatically. The harness can spin up per-worktree instances, capture screenshots and DOM snapshots, and use those signals to validate fixes. The integration is deep but the architecture assumes the agent works in a server-mediated environment, not directly on your machine.  Both tools speak MCP. But the integration philosophies are different enough that Composio’s testing team had to build a custom proxy adapter to get Codex working with Figma and Jira MCPs. When you’re integrating AI coding agents into enterprise toolchains, where the agent needs to read from Jira, push to GitHub, update Slack, and query internal APIs, the implementation depth beneath the protocol matters as much as the protocol itself.  Multi-agent architecture. Claude Code’s Agent Teams spawn multiple sub-agents that each get a dedicated context window with shared task lists and dependency tracking. One sub-agent builds the API while another builds the frontend while a third writes tests, and they can message each other. The Explore sub-tool uses the fast, cheap Haiku model to process large volumes of code and hand relevant context back to Opus for decision-making. This is orchestrated collaboration: a coordinator manages the workflow, and the system is designed to keep a human in the loop as the strategic overseer.  Codex’s multi-agent approach runs each task in its own isolated sandbox. Coordination happens through the codebase, typically via git branches that get merged. OpenAI’s experimental sub-agent support is improving, but Calvin French-Owen notes that “the parallelism still isn’t there quite yet” compared to Claude Code’s more seamless delegation. The tradeoff is that Codex’s isolation model is inherently safer for autonomous operation: agents can’t interfere with each other, can’t access each other’s state, and can’t cascade failures across tasks.  The one-year retrospective from Emergent Minds, one of the most detailed practitioner accounts of Claude Code’s evolution, documents this divergence in real time. The author describes five distinct eras of the tool in twelve months, each making the previous approach look primitive. Community-invented workarounds (ROADMAP.md, ULTRATHINK, SCRATCHPAD.md) were systematically absorbed into native harness features. His meta-observation: “the CLI tooling layer has no moat. Any good pattern gets absorbed into the product.” The harness is evolving fast, and on both sides. The question isn’t which harness is better today. It’s which harness’s evolution trajectory matches where your team is heading. The lock-in nobody’s talking about I want to be careful here, because what I’m about to describe is something I’ve watched happen to my own workflow over the past few months.  Calvin French-Owen’s skill evolution tells you everything about how harness lock-in actually compounds. He started by adding a /commit skill, just telling the model to commit and push in a consistent way. Then he needed agents working in separate worktrees, so he added /worktree. Then he noticed he always planned first and implemented in stages, so he added /implement. Then chaining implement calls was annoying, so he added /implement-all. Then he wasn’t closing the loop on bug feedback from CI, so he added /address-bugs that reads GitHub API comments from Cursor and Codex reviewers and attempts to fix them. Then he realized the whole thing was a loop, so he added /pr-pass, push, wait for CI, address bugs, repeat until clean. He says he wouldn’t have had any success trying to start with this process. It built up over time, each step a small automation that made the next one possible.  That’s six layers of workflow automation, each one built on the previous one, each one specific to Claude Code’s harness architecture, its skill system, its context forking, its sub-agent model. Moving to a different harness doesn’t just mean learning new commands. It means rebuilding that entire compounding chain of automation from scratch in an architecture that may not even support the same abstractions.  Now multiply that by every engineer on a team, every project they touch, every CLAUDE.md file they’ve accumulated, every MCP connector they’ve deployed, every verification protocol they’ve built. That’s the lock-in nobody’s pricing into their decisions.  Claude Code now accounts for roughly 4% of all public GitHub commits, approximately 135,000 per day. SemiAnalysis projects that figure will exceed 20% by the end of 2026. Codex doesn’t have equivalent public data, but developer adoption is clearly substantial and accelerating. Between them, AI coding agents are writing a nontrivial share of the world’s software, and the share is growing on a curve that would have seemed absurd eighteen months ago.  The organizations building workflows around these tools aren’t just adopting a subscription. They’re building institutional knowledge, process documentation, verification protocols, and team habits around a specific agent architecture. Every CLAUDE.md file a team writes. Every Agents.md configuration a team refines. Every MCP connector a team deploys. Every workflow that routes complex architecture tasks to one tool and routine debugging to another. All of it is infrastructure that compounds in value over time, and depreciates to near zero if you switch platforms.  This is precisely analogous to the early cloud wars. In 2010, you could have told an enterprise that AWS and Azure were “basically the same” because they both offered virtual machines and object storage. You would have been technically correct and strategically catastrophic. The organizations that understood the difference between architectures, that grasped how Lambda would reshape application design differently than Azure Functions, how S3’s consistency model would shape data architectures differently than Azure Blob Storage, positioned themselves to build on the right foundation. The organizations that treated cloud as a commodity spent years and millions migrating between platforms when architectural assumptions broke.  We are in the 2010 of AI coding tools. The models look similar in benchmarks. The architectures are diverging along lines that will determine what’s possible in 2028. And procurement decisions are being made by people comparing token costs.  Cursor’s trajectory makes the economics of this divergence concrete. The company hit $1 billion in annual recurring revenue in 24 months and has since doubled to $2 billion ARR, with that run rate doubling in just three months. The team is reportedly still under 200 people. They built on VS Code, which meant zero learning curve for the world’s largest developer community. But here’s the detail that should give every enterprise pause: Cursor reportedly spends 100% of its revenue on AI API costs, most of it flowing to Anthropic, while the company operates at roughly zero gross margin, with total costs slightly exceeding revenue. The company survives on venture capital and the bet that it can either build proprietary models or that API costs will drop fast enough to create margin before the money runs out. Understanding what you’re actually depending on is the point, and that question becomes more expensive to ignore every quarter.  The interchangeability assumption — that any model will do for any task, that switching is a minor inconvenience — is the single most expensive mistake an organization can make in this market. Every piece of evidence from the developer community says the opposite. Developers are building hybrid workflows because no single tool handles everything well. They use Claude Code for initial architecture and feature generation, where contextual depth and verbose reasoning help, and Codex for review, debugging, and routine tasks, where speed and efficiency matter. One developer put it plainly: when working on something complex, he frequently asks Codex to review Claude’s work, and it catches mistakes Claude missed. The tools are complementary, not interchangeable, in the same way that a microscope and a telescope are both optical instruments without being substitutes for each other. What this means for you If you write code for a living, the era of picking one tool and mastering it is ending. The developers extracting the most value right now use both platforms and route work based on what the task needs and how much time they have. Calvin French-Owen’s workflow is representative: Claude Code stays open as a persistent pane for planning, orchestrating his terminal, explaining codebase architecture, and managing git operations, work that benefits from Opus’s creative planning, sub-agent parallelism, and deep tool integration. When he’s ready to actually implement, he flips to Codex, because the code has fewer bugs. He uses Codex and Cursor’s Bugbot for automated code review rather than reading every line of every PR himself, the agents catch subtle bugs better than he does. The skill isn’t using either tool. It’s knowing which harness’s disposition matches the work in front of you.  Track the cost, too. Anthropic’s own data shows the average developer on the Claude Code API spends around $6 per day. Serious daily development frequently hits the Pro plan ceiling and requires the $100 Max tier ($1,200 a year). Codex’s $20 monthly plan goes further for most workflows. Multiple developers report that Codex Pro users almost never hit limits. If your company isn’t paying for your tools, the budget math alone might determine your primary platform. But cheaper isn’t better if you need the other tool’s strengths, and the most productive developers are treating the $20 Codex plan and the $100 Claude plan as complementary investments, not competing subscriptions.  If you lead an engineering team, the decision you’re actually making is not which tool to standardize on. It’s which architectural philosophy to organize around, and whether to build a hybrid workflow that uses both. The procurement lens misses everything that matters. This is process design. How does your team handle task routing? Which work goes to which agent? What are your verification protocols? Do you use one agent to check the other’s work? How do you manage state? Are you investing in CLAUDE.md as persistent project memory, or optimizing for Codex’s clean-start model? How do you handle the security implications of Claude Code’s full-access local execution versus Codex’s sandboxed isolation?  These questions don’t appear in any vendor comparison. They should be in every engineering leader’s decision framework. The organizations that treat this as an architectural commitment rather than a tool selection build on it. The rest spend the next year optimizing last quarter’s workflow.  If you’re a non-technical leader making budget decisions, understand that your team isn’t asking for a tool purchase. They’re asking you to commit to a workbench, a way of organizing the relationship between human engineers and AI agents that will shape your velocity, your security posture, your ability to hire, and your switching costs for years. The right question is not “which tool is cheapest?” It’s “which architectural philosophy matches how our team works, and how much does it cost us to change our mind in eighteen months?”  The answer to the second question is: a lot. And it goes up every quarter, because every quarter your team builds more workflow infrastructure around whichever architecture you chose.  If you’re an AI enthusiast watching this space, the deeper signal isn’t about coding tools at all. It’s about what happens when AI capabilities commoditize while the systems built around them diverge. Language models are converging. The harnesses around them are not. This same pattern will play out in every domain AI touches, legal, medical, financial, creative. The model will become the commodity. The harness will become the product. And the organizations that understand that distinction earliest will build the deepest moats. The question that’s coming SemiAnalysis estimates that Claude Code’s share of GitHub commits will quintuple by year’s end. AI agents aren’t on the margin of software development anymore. They’re becoming the infrastructure.  One complication makes this harder than it looks: the harness layer itself is evolving fast. The author’s conclusion: “the CLI tooling layer has no moat. Any good pattern gets absorbed into the product.” Boris Cherny, head of Claude Code, confirmed that the tool’s own codebase was rewritten every few months, with no code from six months prior remaining in production. OpenAI is moving at a similar pace, shipping the Codex App Server, experimental sub-agent support, and harness engineering documentation in rapid succession.  So you’re choosing an architecture that will itself change dramatically, but whose philosophy will persist. Local-first versus cloud-first, agent-remembers versus codebase-remembers, collaborative versus autonomous. Those are stable commitments even as the implementation details churn.  The question the industry has been asking, which model is best, was always going to have a short shelf life. I’ve spent three weeks watching developers argue about it, and the answer keeps changing every benchmark cycle. The models improve quarterly. Any advantage is temporary. But the question that replaces it, which architecture should I build around, has a much longer half-life, because architectures create dependencies, dependencies compound, and compound dependencies are the hardest thing in technology to undo.  We’re watching two of the most important companies in AI make genuinely different bets about how humans and AI agents should work together. One bet says: the agent belongs in your environment, with full access, deep integration, and collaborative reasoning. The other bet says: the agent belongs in a clean room, isolated, autonomous, and efficient. Both bets are working. Both are winning adherents. Neither is converging toward the other.  The organizations and individuals who understand this, who stop comparing benchmarks and start comparing architectures, who treat the harness decision as the strategic commitment it is, will make the right bets. Everyone else will discover their lock-in the hard way, the same way enterprises discovered cloud lock-in: not in a moment of revelation, but in a budget meeting where the switching cost finally shows up as a line item, and you realize you chose a philosophy, not just a product.
