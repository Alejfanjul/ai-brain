# 1.1 — Modelo de Entidades: App Restaurante HiHotel

> **Módulo:** 1.1 — Entidades, Atributos e Relacionamentos
> **Data:** 2026-04-05
> **Contexto:** Duke Beach Hotel — múltiplos PDVs (restaurante, praia, bar, lago, room service)

---

## Entidades

### PDV (Ponto de Venda)
O conceito mais abrangente. Cada local de venda no hotel é um PDV.

| Atributo | Tipo | Exemplo |
|----------|------|---------|
| id | identificador | pdv_001 |
| nome | texto | "Restaurante Praia" |
| tipo | enum | RESTAURANTE, BAR, PRAIA |
| canais | lista de enum | [PRESENCIAL, ROOM_SERVICE] |
| setores | lista | ["interno", "deck"] |
| horario_canal | objeto | { ROOM_SERVICE: { inicio: "22:30", fim: "07:00" } } |
| ativo | booleano | true |

**PDVs do Duke:**
- Restaurante Praia (setores: interno, deck)
- Serviço de Praia (posições: mesas, cadeiras, guarda-sol)
- Bar Lago
- Lago

> **Room Service:** Não é um PDV separado — é um **canal de atendimento**
> do Restaurante Praia. Compartilha cozinha, estoque e cardápio com o restaurante.
> Funciona o dia todo (durante horário do restaurante = menu completo com restrições
> por categoria de quarto; fora do horário = menu reduzido 22:30–07:00).
> Sem taxa de entrega.

> **Nota de modelagem:** O PDV é o que torna o sistema escalável.
> Sem ele, você codifica "restaurante" direto no sistema e depois
> não consegue reutilizar pro bar. Com ele, qualquer hotel pode
> configurar seus próprios PDVs.

### Local (do Complexo Hoteleiro)
O hotel não é um prédio só — é um complexo com vários edifícios e áreas.
Qualquer ponto do complexo pode ser destino de um pedido.

| Atributo | Tipo | Exemplo |
|----------|------|---------|
| id | identificador | local_001 |
| nome | texto | "Prédio Administrativo" |
| tipo | enum | PREDIO_HOSPEDES, PREDIO_ADMIN, AREA_EXTERNA, OUTRO |
| descricao | texto | "Prédio com escritórios e quarto da direção" |

**Locais do Duke:**
- Prédio principal (quartos 1–33, restaurante, recepção)
- Prédio administrativo (escritórios, quarto do diretor)
- Casa Amarela (prédio secundário, salas de reunião)
- Área externa (praia, lago, piscina)

> **Por que é entidade separada:** Quando o diretor pede comida no
> quarto dele (prédio admin), ou alguém pede na sala de reunião
> da Casa Amarela, o destino não é uma mesa de PDV. É um ponto
> qualquer do complexo. O modelo precisa aceitar isso.
>
> **Conceito novo — Entrega Avulsa:** Pedidos pra locais que não são
> mesas de PDV (sala de reunião, quarto do diretor). O pedido tem
> uma Posição com tipo AVULSO vinculada a um Local em vez de um PDV.

### CategoriaQuarto
Define regras de elegibilidade para room service por tipo de quarto.

| Atributo | Tipo | Exemplo |
|----------|------|---------|
| id | identificador | catq_001 |
| nome | texto | "Premium" |
| quartos | lista | ["1", "2", "3", "4", "5", "6", "7"] |
| categorias_permitidas | lista de enum | [TODAS] |
| descricao_restricao | texto | null (sem restrição) |

**Categorias do Duke:**

| Categoria | Quartos | Pode pedir |
|-----------|---------|------------|
| Premium | 1–7 | Tudo (mesa grande, sem restrição) |
| Standard | 8–33 | Lanches, Entradas, Porções, Saladas, Sobremesas, **Bebidas** |

> **Bebidas:** Permitidas pra todos os quartos, em qualquer horário.
> A restrição dos quartos 8–33 é só sobre pratos que precisam de mesa grande.

> **Regra de elegibilidade:** Quando canal = ROOM_SERVICE, o sistema
> cruza o número do quarto com a CategoriaQuarto pra filtrar o cardápio.
> O hóspede do quarto 25 nem vê os pratos principais no menu.
>
> **Por que é configurável:** No Duke, a restrição é por tamanho da mesa.
> Em outro hotel, pode ser por tipo de diária (all-inclusive vs standard).
> CategoriaQuarto como entidade permite que cada hotel defina suas regras.
>
> **Conceito novo — Regra de Elegibilidade:** Uma restrição que cruza
> atributos de duas entidades (quarto × item) pra determinar o que
> é permitido. Isso é muito comum em sistemas de negócio.

### Posição
Generalização de "mesa" — porque na praia não é mesa, é cadeira/guarda-sol.

| Atributo | Tipo | Exemplo |
|----------|------|---------|
| id | identificador | pos_007 |
| pdv_id | referência → PDV | pdv_001 (null se AVULSO) |
| local_id | referência → Local | local_001 (sempre presente) |
| identificador | texto | "Mesa 7", "Guarda-sol 3", "Sala Reunião 2" |
| tipo | enum | MESA, CADEIRA, GUARDA_SOL, QUARTO, AVULSO |
| capacidade | número | 4 |
| setor | texto | "deck", "piscina" |
| categoria_quarto_id | referência → CategoriaQuarto | catq_002 (só pra quartos) |
| status | enum | LIVRE, OCUPADA, RESERVADA, INTERDITADA |

> **Decisão de modelagem:** Chamamos de "Posição" e não "Mesa"
> porque o sistema precisa funcionar na praia (guarda-sol),
> no room service (quarto) e em qualquer contexto futuro.
> Essa é a diferença entre modelar pra UM caso vs modelar o DOMÍNIO.
>
> **Posição avulsa:** Quando tipo = AVULSO, não pertence a nenhum PDV.
> Exemplos: "Sala Reunião - Casa Amarela", "Quarto Diretor - Admin".
> Vinculada a um Local do complexo. Pode ou não ser permanente.
>
> **Categoria do quarto:** Quando tipo = QUARTO, o campo categoria_quarto_id
> define as regras de elegibilidade pro room service.

### Cardápio
Conjunto de itens disponíveis. Um por enquanto, preparado pra múltiplos.

| Atributo | Tipo | Exemplo |
|----------|------|---------|
| id | identificador | card_001 |
| nome | texto | "Cardápio Principal" |
| tipo | enum | PERMANENTE, SAZONAL, PROMOCIONAL, EVENTO |
| vigencia_inicio | data | 2026-01-01 |
| vigencia_fim | data | null (permanente) |
| pdvs | referência → PDV[] | [pdv_001, pdv_002] |
| ativo | booleano | true |

> **Preparado pra futuro:** tipo SAZONAL cobre combos de Natal,
> PROMOCIONAL cobre happy hour, EVENTO cobre pacotes especiais.
> A vigência permite ativar/desativar automaticamente.

### ItemCardápio
Um prato, bebida ou produto específico.

| Atributo | Tipo | Exemplo |
|----------|------|---------|
| id | identificador | item_001 |
| nome | texto | "Filé ao Molho Madeira" |
| descricao | texto | "Filé mignon grelhado com molho madeira e arroz" |
| preco | decimal | 89.90 |
| categoria | enum | ENTRADA, PRINCIPAL, SOBREMESA, BEBIDA, PETISCO |
| estacao_preparo | referência → Estação | estacao_cozinha |
| disponivel | booleano | true |
| imagem_url | texto | (opcional, pra cardápio digital) |

> **Relação com Cardápio:** N:N — um item pode estar em vários cardápios
> (o filé aparece no cardápio permanente E no combo de Natal).
> Isso se resolve com uma tabela intermediária (CardápioItem) que pode
> ter preço diferente por cardápio.

### Combo
Um agrupamento de itens do cardápio vendido como pacote com preço próprio.

| Atributo | Tipo | Exemplo |
|----------|------|---------|
| id | identificador | combo_001 |
| nome | texto | "Combo Praia" |
| descricao | texto | "2 caipirinhas + porção de batata" |
| preco_combo | decimal | 59.90 |
| itens | referência → ItemCardápio[] | [caipirinha, caipirinha, batata_frita] |
| cardapio_id | referência → Cardápio | card_002 (sazonal verão) |
| ativo | booleano | true |

> **Decisão de modelagem:** Combo é um ItemCardápio "especial" com tipo COMBO.
> Na hora do pedido, ele "explode" nos itens individuais — cada um vira
> um ticket pra sua estação (caipirinha → bar, batata → cozinha).
> Pro cliente, aparece como "Combo Praia R$59,90".
> Pra cozinha/bar, chegam os tickets separados.
>
> **Alternativa descartada:** Itens separados com desconto condicional.
> Seria mais flexível mas muito mais complexo (o sistema precisa detectar
> combinações em tempo real). Combo como item composto é mais simples
> e cobre 90% dos casos.

### ReservaMesa
Reserva antecipada de posição (mesa) no restaurante.

| Atributo | Tipo | Exemplo |
|----------|------|---------|
| id | identificador | res_001 |
| posicao_id | referência → Posição | pos_007 (Mesa 7) |
| pdv_id | referência → PDV | pdv_001 |
| cliente_id | referência → Cliente | cli_001 (opcional) |
| nome_contato | texto | "João Silva" |
| telefone | texto | "+55 48 99999-0000" |
| data_hora | datetime | 2026-04-05 20:00 |
| num_pessoas | número | 4 |
| observacao | texto | "Aniversário, precisa de bolo" |
| status | enum | CONFIRMADA, CANCELADA, NO_SHOW, CONCLUIDA |

> **Conexão com Posição:** Quando uma reserva é confirmada, a Posição
> muda status para RESERVADA. Quando o cliente chega, muda pra OCUPADA.
> Se não aparece (NO_SHOW), volta pra LIVRE após tolerância.

### Estação (de preparo)
Onde os itens são preparados — cozinha, bar, confeitaria.

| Atributo | Tipo | Exemplo |
|----------|------|---------|
| id | identificador | est_001 |
| nome | texto | "Cozinha" |
| tipo | enum | COZINHA, BAR, CONFEITARIA |
| pdv_id | referência → PDV | pdv_001 |

**Estações do Duke:**
- Cozinha (recebe pratos)
- Bar Restaurante (recebe bebidas do restaurante)
- Bar Lago (recebe bebidas do bar lago)

> **Por que é entidade separada:** O garçom registra "1 caipirinha + 1 filé".
> O sistema precisa saber que a caipirinha vai pro bar e o filé pra cozinha.
> Sem Estação, o sistema não consegue rotear os tickets.

### Cliente
Quem está consumindo. Dois tipos com comportamento diferente.

| Atributo | Tipo | Exemplo |
|----------|------|---------|
| id | identificador | cli_001 |
| nome | texto | "João Silva" (opcional) |
| tipo | enum | HOSPEDE, EXTERNO |
| quarto | texto | "204" (só hóspede) |
| reserva_id | referência | (só hóspede — link com PMS) |
| documento | texto | CPF (opcional, pra nota) |

> **Herança:** Hóspede e ClienteExterno são especializações de Cliente.
> Hóspede tem quarto e pode debitar na conta do quarto.
> ClienteExterno paga na hora (dinheiro, cartão, pix).
> Essa distinção é uma **decisão de negócio** que vira **decisão técnica**.

### Pedido
Solicitação de itens feita a partir de uma posição. O pedido É a comanda.

| Atributo | Tipo | Exemplo |
|----------|------|---------|
| id | identificador | ped_001 |
| posicao_id | referência → Posição | pos_007 (Mesa 7) |
| pdv_id | referência → PDV | pdv_001 |
| canal | enum | PRESENCIAL, ROOM_SERVICE, ENTREGA_INTERNA |
| colaborador_id | referência → Colaborador | col_003 (garçom) |
| cliente_id | referência → Cliente | cli_001 (opcional) |
| data_hora_abertura | datetime | 2026-04-05 12:30 |
| data_hora_fechamento | datetime | null (ainda aberto) |
| status | enum | ABERTO, EM_PREPARO, PARCIAL_ENTREGUE, ENTREGUE, CANCELADO |
| historico_posicoes | lista | ["pos_007", "pos_012"] (se transferiu de mesa) |

> **Nota:** Uma posição pode ter vários pedidos ao longo do dia.
> Cada vez que o garçom volta à mesa e registra novos itens,
> pode ser um novo pedido OU itens adicionados ao pedido existente.
> Decisão: tratamos como **pedidos separados** vinculados à mesma conta.
>
> **Transferência de mesa:** O pedido pode mudar de posição.
> O `posicao_id` aponta pra posição ATUAL; `historico_posicoes`
> registra por onde passou. Isso mantém rastreabilidade.
>
> **Room Service:** Quando canal = ROOM_SERVICE, a posição é um Quarto.
> O sistema filtra o cardápio pela CategoriaQuarto do quarto destino.
>
> **Entrega Interna:** Quando canal = ENTREGA_INTERNA, a posição é
> AVULSO — pode ser sala de reunião, quarto do diretor, etc.
> Não tem restrição de cardápio. Cobrança definida caso a caso.

---

### ItemPedido
Um item específico dentro de um pedido. É o "ticket" que aparece na cozinha ou bar.

| Atributo | Tipo | Exemplo |
|----------|------|---------|
| id | identificador | ip_001 |
| pedido_id | referência → Pedido | ped_001 |
| item_cardapio_id | referência → ItemCardápio | item_001 |
| quantidade | número | 2 |
| observacao | texto | "Sem cebola, ponto mal passado" |
| estacao_destino | referência → Estação | est_001 (cozinha) |
| status | enum | AGUARDANDO, PREPARANDO, PRONTO, ENTREGUE, CANCELADO |
| combo_origem_id | referência → Combo | combo_001 (se veio de combo, null se avulso) |
| dividido | booleano | false (preparado pra divisão futura) |
| data_hora_envio | datetime | quando foi pra cozinha |
| data_hora_pronto | datetime | quando a cozinha marcou pronto |

> **Esse é o ticket.** Quando o garçom registra o pedido, cada item
> vira um ticket na estação correspondente. A cozinha vê seus tickets,
> o bar vê os dele. Quando marca "pronto", o garçom é notificado.
>
> **Combo:** Se o item veio de um combo, `combo_origem_id` rastreia a origem.
> Isso permite que na conta apareça "Combo Praia R$59,90" mas na
> cozinha/bar cheguem tickets individuais.
>
> **Divisão (futuro):** O campo `dividido` está preparado pra quando
> 2 pessoas dividirem um prato. Quando ativado, o valor do item
> seria rateado entre as contas individuais dos clientes na mesa.

### Conta
Registro financeiro. Pode ser por posição (mesa inteira) ou por cliente (contas separadas).

| Atributo | Tipo | Exemplo |
|----------|------|---------|
| id | identificador | conta_001 |
| posicao_id | referência → Posição | pos_007 |
| cliente_id | referência → Cliente | cli_001 (se conta individual) |
| tipo | enum | MESA_INTEIRA, INDIVIDUAL |
| forma_pagamento | enum | DINHEIRO, CARTAO, PIX, DEBITO_QUARTO |
| valor_total | decimal | 245.80 |
| gorjeta_percentual | decimal | 10.0 (fixo) |
| gorjeta_valor | decimal | 24.58 (calculado) |
| status | enum | ABERTA, FECHADA, CANCELADA |
| impressa | booleano | false |
| data_fechamento | datetime | quando foi paga |

> **Regra de negócio crítica:** Se tipo = DEBITO_QUARTO,
> o sistema precisa do número do quarto + confirmação do hóspede.
> Isso é integração com o PMS (módulo 2.3 do plano de estudo).
>
> **Gorjeta:** Fixa (10%), calculada por conta. Se a conta é
> MESA_INTEIRA, gorjeta sobre o total. Se INDIVIDUAL, gorjeta
> sobre o consumo daquela pessoa.
>
> **Impressão:** O sistema precisa gerar e enviar pra impressora
> térmica. O campo `impressa` rastreia se já foi impressa
> (evita reimprimir por acidente, mas permite reimpressão manual).

### Colaborador
Quem opera o sistema — garçom, barman, cozinheiro.

| Atributo | Tipo | Exemplo |
|----------|------|---------|
| id | identificador | col_001 |
| nome | texto | "Stella" |
| funcao | enum | GARCOM, BARMAN, COZINHEIRO, MAITRE, GERENTE |
| nivel_hierarquico | número | 1–5 (1=cumin, 5=gerente) |
| pdvs_autorizados | referência → PDV[] | [pdv_001, pdv_002] |
| permissoes | lista de enum | [REGISTRAR_PEDIDO, DAR_DESCONTO, CANCELAR_ITEM, ...] |
| turno | enum | MANHA, TARDE, NOITE |
| ativo | booleano | true |

> **Conexão com SID:** No futuro, o Colaborador no app restaurante
> é o mesmo Colaborador do SID. Mesma pessoa, contextos diferentes.
> Isso é um exemplo de como a ontologia unifica os sistemas.
>
> **Hierarquia de permissões:** Nem todo colaborador pode tudo.
> Cumin (nível 1) registra pedidos. Líder (nível 3) pode dar desconto.
> Gerente (nível 5) pode cancelar itens já enviados e alterar preços.
> Isso é modelado via `nivel_hierarquico` + `permissoes`.
>
> | Nível | Função típica | Pode fazer |
> |-------|--------------|------------|
> | 1 | Cumin | Registrar pedidos |
> | 2 | Garçom | Registrar + transferir mesa |
> | 3 | Líder | Tudo acima + dar desconto |
> | 4 | Maître | Tudo acima + cancelar item enviado |
> | 5 | Gerente | Tudo (incluindo alterar preços, fechar caixa) |

## Mapa de Relacionamentos

```
Local (1) ◄──────── (N) Posição ◄────── (N) ReservaMesa
                      │    │
              (N:1)   │    │ (N:1, se quarto)
                ▼     │    ▼
               PDV    │  CategoriaQuarto ──regra──▶ [categorias permitidas]
                      │
                      │ (1)
                      ▼
                 Pedido (N) ──canal──▶ PRESENCIAL | ROOM_SERVICE | ENTREGA_INTERNA
                  │  │
            (N)   │  │  (N:1)
      ┌───────────┘  └──────────┐
      ▼                         ▼
 ItemPedido (N)            Colaborador ──nível──▶ permissões hierárquicas
   │   │   │
(N:1)  │   │(N:1)
   ▼   │   ▼
ItemCardápio  Estação
   │   │
(N:N)  │(N:1, opcional)
   ▼   ▼
Cardápio  Combo ──contém──▶ ItemCardápio[]

Posição (1) ───────── (N) Conta ──impressão──▶ 🖨️ (rede local)
                             │
                        (N:1)│
                             ▼
                          Cliente
                       (HOSPEDE | EXTERNO)

PDV (1) ◄───────────── (N) Estação
```

**Leitura do mapa:**
- Um Local (prédio/área) contém N posições
- Uma posição pode pertencer a um PDV (mesa, cadeira) ou ser avulsa (sala, quarto admin)
- Uma posição tipo QUARTO tem uma CategoriaQuarto que define elegibilidade de cardápio
- Uma posição gera N pedidos ao longo do dia
- Uma posição pode ter N reservas (em datas/horários diferentes)
- Um pedido tem um canal (presencial, room service, entrega interna)
- Um pedido contém N itens de pedido
- Cada item de pedido referencia 1 item do cardápio
- Cada item de pedido pode vir de 1 combo (opcional)
- Cada item de pedido vai pra 1 estação (cozinha ou bar)
- Um pedido é atendido por 1 colaborador (com permissões hierárquicas)
- Um pedido pode ser transferido de posição (historico_posicoes)
- Uma posição pode gerar N contas (contas separadas)
- Uma conta pertence a 1 cliente específico (ou à mesa inteira)
- Uma conta tem gorjeta fixa (10%) e pode ser impressa (rede local)
- Um cardápio contém N itens (e um item pode estar em N cardápios)
- Um combo agrupa N itens do cardápio com preço único

## Decisões de Produto Registradas

| # | Decisão | Impacto no modelo |
|---|---------|-------------------|
| 1 | Conta pode ser por mesa ou por pessoa | Conta tem tipo (MESA_INTEIRA / INDIVIDUAL) + cliente_id opcional |
| 2 | Hóspede debita no quarto, externo paga na hora | Cliente tem tipo (HOSPEDE / EXTERNO) + formas de pagamento diferentes |
| 3 | Cardápio único por ora, preparado pra múltiplos | Cardápio é entidade separada com tipo e vigência; relação N:N com itens |
| 4 | Sistema substitui comanda física | Pedido É a comanda digital |
| 5 | Cozinha e bar recebem tickets próprios | Estação como entidade + ItemPedido roteado por estação |
| 6 | Múltiplos PDVs (restaurante, praia, bar, lago, room service) | PDV como entidade central; Posição generaliza mesa/cadeira/quarto |

## Conceitos Aprendidos neste Módulo

| Conceito | Definição | Exemplo neste modelo |
|----------|-----------|---------------------|
| **Entidade** | Algo com identidade única no domínio | Mesa, Pedido, Cliente |
| **Atributo** | Propriedade que descreve uma entidade | nome, preço, status |
| **Relacionamento** | Conexão entre duas entidades | Pedido CONTÉM ItemPedido |
| **Cardinalidade** | Quantos de cada lado do relacionamento | 1:N = uma mesa, vários pedidos |
| **Herança** | Especialização (um tipo de) | Hóspede é um tipo de Cliente |
| **Generalização** | Conceito mais amplo que cobre vários casos | Posição generaliza Mesa, Cadeira, Quarto |
| **Decisão de produto → decisão técnica** | Regras de negócio moldam a estrutura | "Conta separada" → Conta tem cliente_id |
| **Canal vs PDV** | Canal é modo de operação dentro de um PDV | Room Service é canal do Restaurante, não PDV separado |
| **Composição (Combo)** | Entidade que agrupa outras | Combo "contém" itens do cardápio mas "explode" no pedido |
| **Rastreabilidade** | Registrar mudanças pra auditoria | historico_posicoes no Pedido, impressa na Conta |
| **Regra de Elegibilidade** | Restrição que cruza atributos de 2 entidades | CategoriaQuarto × ItemCardápio = o que pode pedir |
| **Hierarquia de Permissões** | Diferentes níveis de acesso por função | Cumin não dá desconto, Líder pode |
| **Complexo vs PDV vs Posição** | Três níveis de granularidade espacial | Local (prédio) > PDV (restaurante) > Posição (mesa) |
